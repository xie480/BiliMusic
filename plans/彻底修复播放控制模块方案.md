# 彻底修复播放控制模块方案

## 根因分析（Root Cause Analysis）

### 问题一：冷启动意外自动播放

**现象**：应用在记录退出前的播放进度后，执行冷启动或重启会触发自动播放。

**根因**：`setupPlayer()` 的恢复路径中存在级联副作用。

1. [`setupPlayer()`](src/services/trackPlayer.ts:24) 调用 [`TrackPlayer.skip(startIndex)`](src/services/trackPlayer.ts:85) 恢复播放位置
2. `skip()` 触发 [`PlaybackActiveTrackChanged`](src/services/trackPlayer.ts:491) 事件
3. 事件处理器调用 [`lazyResolve(e.index, false)`](src/services/trackPlayer.ts:530)（`autoPlay=false`）
4. `lazyResolve` 内部的**平滑替换策略**（[`add → skip → remove`](src/services/trackPlayer.ts:403-413)）调用 `skip(actualIndex + 1)`
5. 这个 `skip()` **再次触发** `PlaybackActiveTrackChanged` 事件
6. 第二次事件的处理器检查 [`isCurrentlyPlaying`](src/services/trackPlayer.ts:527)：在特定时序下，播放器状态可能短暂变为 `Playing`/`Buffering`，导致 `autoPlay = true`
7. 级联继续：每个 `skip` → 事件 → `lazyResolve` → `skip` → 事件 → ...

**关键缺陷**：
- [`setupPlayer` 对 `lazyResolve` 的调用是 fire-and-forget](src/services/trackPlayer.ts:95)（不 await），因此无法控制级联
- 在 [`App.tsx:105`](src/App.tsx:105) 中 `setupPlayer()` 也没有被 `await`
- 如果 [`lastPosition` 为 0 或 undefined](src/services/trackPlayer.ts:87)，`pause()` 根本不会执行，`skip()` 后播放器可能自动开始播放

### 问题二：FoldersScreen 交互卡死 + API 请求风暴

**现象**：在 FoldersScreen 点击视频后，触发数十甚至上百次 B站 API 请求（`audioService.getInfo`），页面完全卡死。

**根因**：`lazyResolve` 的平滑替换策略 + `PlaybackActiveTrackChanged` 事件处理器的预加载逻辑形成了**递归级联**。

完整执行链如下：

```
用户点击视频
  ↓
loadQueue(100 tracks)                 [FoldersScreen.tsx:247]
  → TrackPlayer.reset()
  → addTracksBatched(100个占位轨道)
  → _pendingPlay = true
  → TrackPlayer.skip(startIndex)       ← 触发事件 #1
  → lazyResolve(startIndex+1, false)  ← fire-and-forget 预加载
  ↓
事件 #1: PlaybackActiveTrackChanged(e.index=S)
  ↓ _pendingPlay=true → autoPlay=true
  ↓ lazyResolve(S, true)
  ↓   audioService.getInfo()            → API请求 #1
  ↓   平滑替换: add(真实URL, S+1) → skip(S+1) → remove(S)
  ↓     ↑ skip(S+1) 触发事件 #2
  ↓     ↑ TrackPlayer.play() (因为 autoPlayActive=true)
  ↓
事件 #2: PlaybackActiveTrackChanged(e.index=S+1)
  ↓ _pendingPlay=false, isCurrentlyPlaying=true
  ↓ autoPlay = false || true = true
  ↓ lazyResolve(S+1, true) → 真实URL → 立即返回
  ↓ 预加载: lazyResolve(S+2, false)
  ↓   audioService.getInfo()            → API请求 #2
  ↓   平滑替换: add → skip(S+3) → remove
  ↓     ↑ skip(S+3) 触发事件 #3
  ↓
事件 #3: PlaybackActiveTrackChanged(e.index=S+3)
  ↓ 同事件 #2 模式
  ↓ autoPlay = true
  ↓ lazyResolve → 返回 → 预加载 lazyResolve(S+4, false)
  ↓   audioService.getInfo()            → API请求 #3
  ↓   skip(S+5) → 事件 #4 ...
  ↓
  ... 级联继续，直到队列末尾
  = 大约 50 次 API 请求（每2个轨道触发1次）
```

**级联的数学规律**：每 2 个轨道触发 1 个 API 请求 + 1 次 `skip` + 1 次 `PlaybackActiveTrackChanged` 事件。对于 100 轨道的队列，大约产生 **50 次 API 请求**。

**根因代码**：

1. [事件预加载逻辑](src/services/trackPlayer.ts:534-536)：每次 `PlaybackActiveTrackChanged` 都要预加载下一首
2. [平滑替换策略](src/services/trackPlayer.ts:403-413)：`add` → `skip` → `remove` 中的 `skip` 触发新事件
3. 两者结合形成自复制循环：事件 → 预加载 → `lazyResolve` → `skip` → 事件 → 预加载 → `lazyResolve` → `skip` → ...

## 修复方案

### 核心原则

1. **消除递归级联**：打破「事件 → 预加载 → skip → 事件」的自复制循环
2. **消除平滑替换**：不再使用 `add→skip→remove` 策略，改用原地更新
3. **统一播放触发模式**：不再依赖 `_pendingPlay` 事件标志，统一使用显式 `TrackPlayer.play()`
4. **冷启动安全**：确保初始化完成后播放器处于明确的暂停状态

### 修复 A：重写 `lazyResolve` — 放弃平滑替换

**文件**：[`src/services/trackPlayer.ts`](src/services/trackPlayer.ts:307)

**问题**：`add(真实轨道, index+1)` → `skip(index+1)` → `remove(index)` 中的 `skip` 触发 `PlaybackActiveTrackChanged` 事件，是级联的根源。

**方案**：改为**移除占位符后追加真实轨道，不改变活跃索引**。

```typescript
// 新的实现：
async function lazyResolve(index: number, autoPlayActive: boolean = false) {
  if (resolving.has(index)) return;
  resolving.add(index);
  let bvid = '';
  let isActiveTrack = false;
  try {
    const activeIdx = await TrackPlayer.getActiveTrackIndex();
    if (activeIdx === index) {
      isActiveTrack = true;
      usePlayerStore.getState().setResolving(true);
    }
    const queue = await TrackPlayer.getQueue();
    const t = queue[index];
    if (!t || !String(t.url).startsWith('placeholder://')) return;
    
    // ... 获取音频信息（API 调用）保持不变 ...
    // ... 多P展开逻辑保持不变 ...
    
    // 【修复】不再使用平滑替换（add→skip→remove）
    // 改为：先删除占位符，再在相同位置插入真实轨道
    const insertIndex = actualIndex;  // 占位符的当前位置
    await TrackPlayer.remove(actualIndex);  // 移除占位符
    await TrackPlayer.add(newTrack, insertIndex);  // 在相同位置插入真实轨道
    
    // 如果当前播放的就是这个轨道，需要跳转到新插入的轨道
    if (isActiveTrack) {
      await TrackPlayer.skip(insertIndex);
      if (autoPlayActive) {
        await TrackPlayer.play();
      }
    }
  } catch (error) {
    // ... 错误处理保持不变 ...
  } finally {
    if (isActiveTrack) {
      usePlayerStore.getState().setResolving(false);
    }
    resolving.delete(index);
  }
}
```

**变更效果**：
- `TrackPlayer.remove()` 后跟 `TrackPlayer.add()` 不会改变活跃索引（除非移除的就是活跃轨道）
- 需要跳转的情况仅在 `isActiveTrack = true` 时发生（当前播放的恰好是占位符）
- 跳转后可能触发 `PlaybackActiveTrackChanged`，但这是单次事件，不会级联

**风险**：移除占位符后立即 `add` 可能导致短暂的 UI 闪烁（Track 短暂消失）。但由于操作是连续 `await` 的，在 JS 层面几乎是瞬时的，用户无感知。

### 修复 B：重构 `PlaybackActiveTrackChanged` 事件处理器 — 移除自动预加载

**文件**：[`src/services/trackPlayer.ts`](src/services/trackPlayer.ts:491)

**问题**：事件处理器在解析当前轨道后自动预加载下一首，且该预加载会触发更多事件。

**方案**：事件处理器只负责**解析当前轨道**，不再自动预加载下一首。预加载由 `loadQueue` 等调用方显式触发。

```typescript
TrackPlayer.addEventListener(Event.PlaybackActiveTrackChanged, async (e) => {
    // ... 更新 store（bvid、cid 等）保持不变 ...
    
    if (e.index !== undefined) {
      // 【修复】不再检查 _pendingPlay 标志
      // 不再依赖 isCurrentlyPlaying 状态来判断自动播放
      // 统一由调用方（loadQueue、VideosScreen.playFrom 等）显式调用 TrackPlayer.play()
      await lazyResolve(e.index, false);
      // 【修复】移除自动预加载！不再调用 lazyResolve(e.index + 1, false)
      // 预加载改为由 loadQueue 显式管理
    }
    
    if (e.lastTrack?.id) autoCache(e.lastTrack.id as string);
});
```

**变更效果**：
- 事件处理器不再触发任何级联事件
- 每次用户 `skip`（切歌）时只解析当前轨道
- 预加载由 `loadQueue` 在加载队列后主动触发（只触发一次，不级联）

### 修复 C：重写 `loadQueue` — 统一显式 `play()` 模式

**文件**：[`src/services/trackPlayer.ts`](src/services/trackPlayer.ts:106)

**问题**：`_pendingPlay` 标志机制复杂且容易导致时序问题。`FoldersScreen` 不调用 `TrackPlayer.play()` 而是依赖事件标志，与 `VideosScreen` 的显式调用模式不一致。

**方案**：移除 `_pendingPlay` 标志，在所有调用点统一使用显式 `TrackPlayer.play()`。

```typescript
export async function loadQueue(videos: FavoriteVideo[], startBvid?: string) {
  if (!videos || videos.length === 0) return;
  await TrackPlayer.reset();
  const startIndex = Math.max(0, videos.findIndex((v) => v.bvid === startBvid));
  
  const tracks = videos.map((v) => ({
    id: v.bvid,
    url: `placeholder://${v.bvid}`,
    title: v.title,
    artist: v.upper?.name || '未知作者',
    artwork: v.cover,
    duration: v.duration,
  }));
  await addTracksBatched(tracks);
  await TrackPlayer.skip(startIndex);
  // 【修复】不再设置 _pendingPlay 标志
  // 由调用方在 loadQueue 完成后显式调用 TrackPlayer.play()
}
```

### 修复 D：加固 `setupPlayer` 冷启动恢复路径

**文件**：[`src/services/trackPlayer.ts`](src/services/trackPlayer.ts:24)

**问题**：`lazyResolve` 的 fire-and-forget 调用在后台产生级联，且 `pause()` 可能因时序问题未能阻止播放。

**方案**：在 `setupPlayer` 完成后显式确保播放器暂停，并清理模块级状态。

```typescript
export async function setupPlayer() {
  // ... 初始化代码保持不变 ...
  
  const store = usePlayerStore.getState();
  if (store.queue && store.queue.length > 0) {
    const tracks = store.queue.map((v) => ({/* ... */}));
    await addTracksBatched(tracks);
    
    const startIndex = Math.max(0, store.queue.findIndex((v) => v.bvid === store.currentBvid));
    await TrackPlayer.skip(startIndex);
    
    const lastPosition = storage.getNumber('lastPlaybackPosition');
    if (lastPosition && lastPosition > 0) {
      await TrackPlayer.seekTo(lastPosition);
    }
    
    // 【修复】无论 lastPosition 是否存在，都确保暂停
    await TrackPlayer.pause();
    
    // 【修复】只解析当前轨道，不触发级联
    lazyResolve(startIndex, false).catch(() => {});
  }
  
  // 【修复】重置 _pendingPlay 标志，防止从之前的状态泄露
  _pendingPlay = false;
  
  _ready = true;
}
```

### 修复 E：冷启动后移除 `lazyResolve` 的 Fire-and-Forget 预加载

**文件**：[`src/services/trackPlayer.ts`](src/services/trackPlayer.ts:128-130)

**问题**：`loadQueue` 末尾的 `lazyResolve(startIndex + 1, false)` 在后台异步执行，一旦完成其平滑替换，就会触发级联事件链。

**方案**：移除该 fire-and-forget 预加载调用。新的事件处理器（修复 B 之后）不再触发级联，因此即使保留也无害，但为了清晰和安全，一并移除。

```typescript
export async function loadQueue(videos: FavoriteVideo[], startBvid?: string) {
  // ... 代码 ...
  await addTracksBatched(tracks);
  await TrackPlayer.skip(startIndex);
  // 【修复】移除 lazyResolve(startIndex + 1, false) 的 fire-and-forget 调用
}
```

## 修复后的事件流对比

### 修复前（当前有缺陷的流程）：

```
loadQueue(100 tracks)
  → skip(startIndex)
  → 事件 #1: lazyResolve(S, true) → add→skip(S+1)→remove(S)
    → 事件 #2: lazyResolve(S+1) → 返回 → 预加载 lazyResolve(S+2, false)
      → skip(S+3) → 事件 #3 → 预加载 lazyResolve(S+4, false)
        → skip(S+5) → 事件 #4 → ... (级联到队列末尾)
  → lazyResolve(startIndex+1, false) [fire-and-forget，重复解析同一索引]
```

**结果**：~50 次 API 请求 + 播放器状态混乱 + 页面卡死

### 修复后（稳定流程）：

```
loadQueue(100 tracks)
  → skip(startIndex)
  → 事件 #1: lazyResolve(S, false) → remove(S) → add(真实URL, S)
    → 如果 isActiveTrack: skip(S) [单次事件，无级联]
  → TrackPlayer.play() [调用方显式触发]
  → navigate('Player')
```

**结果**：1 次 API 请求（当前轨道）+ 无级联 + 播放器状态明确 + 页面正常跳转

## 预期的副作用与风险

1. **无平滑替换 = 可能有短暂闪烁**：当当前播放的轨道是占位符时（极端情况），移除占位符再添加真实轨道的操作可能导致 Track 对象短暂消失。但 `remove` + `add` 是连续 `await` 的，在实际测试中这是不可感知的。

2. **无事件级联预加载 = 切歌时可能稍慢**：当前修复取消了事件驱动的自动预加载。这意味着用户切歌到下一首时，需要等待 `lazyResolve` 完成 API 请求后才能播放。但由于 `loadQueue` 可以主动触发预加载（如 `lazyResolve(startIndex + 1, false)`），且事件处理器不再级联，所以总 API 请求数大幅降低（从 ~50 次降至 1-2 次）。

3. **与 VideosScreen 一致的播放模式**：`FoldersScreen` 现在使用与 `VideosScreen` 完全相同的 `loadQueue` + `TrackPlayer.play()` 模式。这消除了不一致的行为，降低了未来调试的复杂度。
