# BiliMusic 全面性能深度优化方案

## 一、 音频加载机制优化 (播放界面)

### 1. 诊断与瓶颈分析
当前架构采用 `placeholder://` 占位符机制，在 `PlaybackActiveTrackChanged` 事件触发后，才通过 `lazyResolve` 发起网络请求获取真实音频流。这导致了**“点击播放 -> 占位符加载 -> 触发事件 -> 网络请求 -> 真实音频加载 -> 发声”**的超长链路，初始加载耗时严重。

### 2. 优化架构设计
*   **首屏秒开 (First-Track Pre-resolution)**: 拦截播放动作，在将队列推入 `TrackPlayer` 之前，**并发**请求首支单曲的真实 URL。利用 React Navigation 的转场动画时间掩盖网络请求耗时。
*   **滑动窗口预加载 (Sliding Window Preload)**: 废弃被动的事件驱动加载，改为主动的后台预加载。维护一个大小为 2 的预加载窗口，当播放第 N 首时，后台静默解析 N+1 和 N+2 首的 URL 并替换占位符。
*   **CDN 路由全局缓存 (Global CDN Routing Cache)**: `selectFastestUrl` 目前针对单视频测速。B站音频 CDN 域名池有限（如 `upos-sz-mirrorcos.bilivideo.com` 等），应实现**域名级测速缓存**，跳过重复的 HEAD 请求。
*   **I/O 调度优化**: 将音频解析任务放入自定义的 `TaskQueue`，赋予不同优先级（当前播放=High，预加载=Low），避免并发请求阻塞主线程网络 I/O。

## 二、 路由与页面切换性能瓶颈 (VideosScreen -> FoldersScreen)

### 1. 深度排查维度
*   **内存泄漏与繁重状态同步**: `VideosScreen` 挂载了大量 `FlatList` 节点（`MAX_QUEUE_SIZE = 200`）。返回时，React 需同步卸载大量 Fiber 节点，阻塞 JS 线程。
*   **页面生命周期管理**: 导航动画（约 300ms）与组件卸载/重渲染重叠，导致掉帧（UI 主线程阻塞）。
*   **资源释放不及时**: 图片资源（FastImage）在列表快速滑动或页面销毁时未及时回收。

### 2. 修复逻辑与重构方案
*   **FlatList 极致调优**:
    *   强制启用 `removeClippedSubviews={true}` (卸载屏幕外视图)。
    *   调优 `windowSize={5}` (默认 21 过大)，`maxToRenderPerBatch={10}`，`initialNumToRender={10}`。
*   **组件级 Memoization**: 将 `VideosScreen` 和 `FoldersScreen` 的 `renderItem` 抽离为独立的 `React.memo` 组件，并自定义 `areEqual` 函数，彻底阻断无关状态引发的重渲染。
*   **转场动画避让 (InteractionManager)**: 在 `VideosScreen` 的 `useEffect` 清理函数中，使用 `InteractionManager.runAfterInteractions` 延迟执行繁重的状态清理工作。
*   **启用 React Native Screens 冻结**: 确保 `react-native-screens` 开启了 `freezeOnBlur`，使 `FoldersScreen` 在被覆盖时完全停止渲染周期。

## 三、 全局视图渲染优化

### 1. 核心策略
*   **减少过度绘制 (Overdraw)**: 
    *   排查并移除冗余的 `backgroundColor`。例如，如果父容器已有不透明背景，子容器应避免设置相同的背景色。
    *   毛玻璃组件 (`GlassView`) 极其消耗 GPU，需限制其使用范围，避免在长列表中滥用。
*   **压缩视图层级**: 
    *   重构 `ListItem` 等高频组件，使用 `React.Fragment` 或扁平化布局，消除无意义的 `<View>` 嵌套。
*   **精细化组件局部刷新**: 
    *   严格审查 Zustand store 的 `useStore` 调用。必须使用 Selector 模式（如 `usePlayerStore(s => s.currentBvid)`），防止 Store 任意属性变更导致全量重渲染。
*   **复杂计算移至工作线程**: 
    *   `FoldersScreen` 中的全局搜索过滤 (`filteredVideos`) 在数据量大时会阻塞 JS 线程。应引入 `useMemo` 缓存计算结果，或使用 `lodash/debounce` 防抖输入。
*   **底层光栅化性能提升**: 
    *   对于静态且复杂的 UI 块（如不包含动画的卡片背景），在 Android 启用 `renderToHardwareTextureAndroid={true}`，iOS 启用 `shouldRasterizeIOS={true}` 配合 `rasterizationScale`，将视图缓存为 GPU 纹理。
