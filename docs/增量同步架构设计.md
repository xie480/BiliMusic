# BiliMusic 收藏夹增量同步架构设计方案

## 1. 核心思路与背景
当前系统在同步收藏夹时采用全量拉取策略，随着用户收藏视频数量的增加，全量同步会导致严重的性能瓶颈（网络请求多、耗时长、容易触发 B 站风控限流）。
由于 B 站收藏夹接口（`/x/v3/fav/resource/list`）默认按**收藏时间倒序**（`order='mtime'`）返回数据，最新添加的视频始终在第一页。基于此特性，我们可以采用**基于游标（Cursor）和状态标记**的增量同步方案：只需从第一页开始拉取，直到遇到上次同步记录的“游标”即可停止，从而彻底避免全量开销。

## 2. 状态标记设计 (Sync Metadata)
为追踪数据变化，我们需要在本地持久化存储中为每个收藏夹维护一份同步元数据（`FolderSyncMeta`）。

```typescript
export interface FolderSyncMeta {
  folderId: number;
  lastSyncTime: number;      // 上次成功同步的时间戳
  latestBvid: string | null; // 上次同步时该收藏夹最新的视频 BVID（游标）
  mediaCount: number;        // 上次同步时该收藏夹的视频总数
}
// 存储结构示例：Record<number, FolderSyncMeta>
```

## 3. 增量同步业务逻辑
当触发同步操作（如启动 App、下拉刷新、导入新收藏夹）时，执行以下流程：

### 3.1 识别增量与目标文件夹
1. 调用 `getFavoriteFolders` 获取用户当前所有可见收藏夹列表。
2. 遍历文件夹列表，读取本地存储的 `FolderSyncMeta`。
3. **判断同步策略**：
   - **全新文件夹**：本地无 `FolderSyncMeta`，需执行**单文件夹全量同步**。
   - **有更新的文件夹**：当前 `media_count` > 本地 `mediaCount`，执行**增量同步**。
   - **无变化的文件夹**：当前 `media_count` == 本地 `mediaCount`，直接跳过（除非用户强制全量同步）。

### 3.2 增量拉取策略 (Stop Condition)
对于需要增量同步的文件夹，按页（`pn=1, 2, ...`）请求视频列表：
1. 遍历当前页返回的视频列表。
2. **终止条件匹配**：如果遇到视频的 `bvid` 等于本地记录的 `latestBvid`，说明接下来的视频在上次同步时已经拉取过，**立即终止该文件夹的后续翻页请求**。
3. 收集所有在遇到 `latestBvid` 之前遍历到的新视频，作为该文件夹的增量数据。

## 4. 数据去重与一致性保障
全局索引（`globalIndex`）维护了所有视频的去重集合。增量数据合并时需遵循 **Upsert（更新或插入）** 原则：

1. **主键去重**：以 `bvid` 作为唯一主键。
2. **多文件夹归属处理**：同一个视频可能存在于多个收藏夹中。
   - 如果 `globalIndex` 中不存在该 `bvid`，则直接插入，并初始化 `folderIds = [currentFolderId]`。
   - 如果已存在，则更新其元数据（如 title, cover 可能有变），并将 `currentFolderId` 追加到 `folderIds` 数组中（需去重）。
3. **原子性写入**：在内存中完成所有增量数据的 Merge 后，一次性将更新后的 `globalIndex` 序列化写入本地 Storage，避免中间状态导致数据不一致。

## 5. 删除与修改操作的兼容策略
B 站接口不提供删除事件的回调，增量同步默认无法感知历史数据的删除或失效。

### 5.1 软删除检测与补偿
通过比对 `mediaCount` 发现删除操作：
- 设本次拉取到的新增视频数为 `N`。
- 如果 `当前接口 media_count < 本地 mediaCount + N`，说明用户在 B 站删除了视频，或者有视频失效被系统清理。
- **处理策略**：
  - **轻量级容忍**：增量同步阶段**不主动处理删除**。已删除的视频仍保留在本地索引中，当用户尝试播放该失效视频时，播放器捕获异常并标记该视频为“失效（attr != 0）”，UI 层置灰。
  - **异步补偿（推荐）**：当检测到数量不匹配时，将该文件夹标记为 `dirty`。在系统空闲时（或下次同步时），对该文件夹触发一次**单文件夹全量校准同步**，以清理本地已删除的脏数据。

### 5.2 视频属性修改
视频标题、封面等元数据的修改通常不影响播放。增量同步拉取到的新数据会覆盖旧数据。对于未拉取到的历史数据，可在播放时通过 `getVideoInfo` 接口实时获取最新元数据并更新本地缓存。

## 6. 容错恢复与中断处理
移动端网络环境复杂，同步过程随时可能中断（如杀后台、断网）。

1. **事务性元数据更新**：
   - `FolderSyncMeta` 的更新必须在**增量数据成功写入 `globalIndex` 之后**进行。
   - 如果同步中断，`FolderSyncMeta` 不会更新。下次同步时，系统仍会使用旧的 `latestBvid` 作为游标，重新拉取并覆盖合并（Upsert 是幂等的），不会产生重复数据或遗漏。
2. **细粒度任务控制**：
   - 将同步任务拆分为“单文件夹同步子任务”。
   - 某个文件夹同步失败，不影响其他已成功文件夹的元数据更新和数据持久化。
3. **重试机制**：保留现有的指数退避重试逻辑（`executeWithBackoff`），应对 B 站接口的临时限流（412/429）。
